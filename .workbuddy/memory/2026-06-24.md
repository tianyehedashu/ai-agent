## 生图模型调不通 — 根因排查

用户反馈"数据库已配置生图模型，但调不通/测试有问题"。排查后定位为**系统模型凭据断层**（架构性问题，非测试 bug）。

### 三条生图路径与凭据来源
1. **Gateway 代理** `/api/v1/openai/v1/images/generations`（openai_compat_router → ProxyUseCase.image_generation）：✅ 正确从数据库凭据解析，火山走 `volcengine_direct_image_generation` 直连，endpoint_id 取自凭据 `extra.image_endpoint_id`。
2. **Listing Studio 8图任务**（listing_studio_router `/image-gen` → ProductImageGenTaskUseCase → ImageGenerator）：⚠️ 系统模型凭据丢失。
3. **对话生图**（chat_image_gen `_run_image_gen_mode` → ImageGenerator）：⚠️ 同样系统模型凭据丢失。

### 根因
`ChatModelResolutionUseCase._resolve_visible_system_image_gen`（chat_model_resolution_use_case.py:375-390）解析**系统模型**时返回 `ResolvedImageGenModel(provider, model=real_model, is_system=True)`，**api_key=None, api_base=None**。

下游 `ImageGenerator._generate_volcengine` 在 `api_key_override=None` 时回退到全局 `settings.volcengine_api_key`（环境变量），而非数据库凭据。若环境变量未配 VOLCENGINE_API_KEY → 报"VOLCENGINE_API_KEY 未配置"；即便配了 api_key，`model` 字段是 litellm 模型名（如 `seedream`）而非方舟 `ep-xxx` endpoint_id，火山方舟生图必须用 endpoint_id 作 model，传错会 400/404。

**对比**：个人模型走 `_resolve_personal_image_gen`，能正确从数据库带出 api_key/api_base，所以个人模型可通、系统模型不通。

### 关键文件
- `backend/domains/agent/application/chat_model_resolution_use_case.py:375-390`（系统模型解析丢凭据）
- `backend/domains/agent/infrastructure/llm/image_generator.py:126-163`（回退环境变量）
- `backend/domains/gateway/domain/policies/volcengine_image.py`（火山需 ep-xxx 作 model）
- `backend/domains/gateway/application/proxy_non_chat_pipeline.py:141`（正确路径参考）

### 测试盲区
`test_product_image_gen_reference_chain.py` 等 mock 了 ImageGenerator，未覆盖"系统模型→api_key_override=None→环境变量回退"分支，故测试通过但生产调不通。

- 模型1探活失败 reason="凭据 extra.image_endpoint_id 为空"；模型2-4 `test=None`（未测）

### 两个独立问题
1. **DB 配置断层**（探活失败根因）：团队凭据 `extra` 缺 `image_endpoint_id` → 探活失败 → `last_test_status=failed` → `is_connectivity_requestable=False` → 模型被 `list_visible_models` 排除 → 对话侧 `allowed_image_gen_system_ids` 为空 → 报"无可用图像生成模型"
2. **ImageGenerator 运行时 bug**（即使模型可选也调不通）：`_generate_volcengine` 用 `payload["model"] = model or endpoint_id`，`_resolve_personal_image_gen` 返回的 `model` 是 bare model name（如 `doubao-seedream-3-0-t2i-250415`），永远非 None → endpoint_id 被遮蔽 → 火山 `/images/generations` 需 `ep-xxx` 作 model → 400/404。已用脚本验证确认。

### 修复方向
1. DB：给团队凭据 `huoshan-jmy` 的 `extra` 补 `image_endpoint_id=ep-m-20251224180807-6jpr7`，再重测 4 个模型探活
2. 代码：`ImageGenerator._generate_volcengine` 火山分支应优先用 `endpoint_id` 作 model（对齐 probe 路径 `build_volcengine_image_request`）；或在 `_resolve_personal_image_gen` 对火山返回 `model=None` 让 endpoint_id 生效

## 架构诊断（用户追问的 4 点）

### 1. 个人 vs 团队路径差异根因
- `GatewayProxyProtocol` 只有 `chat_completion`/`embedding`，**无 image_generation**
- chat text 走 `AgentLlmFacade → gateway_proxy.chat_completion`（只传虚拟模型名，凭据由 Gateway 解析）✅
- image_gen 绕过 Gateway，走 `ImageGenerator` 直连 ❌
- 个人模型：`_resolve_personal_image_gen` 调 `resolve_registered_model` 查 DB 凭据 → 带 api_key/api_base
- 系统/团队模型：`_resolve_visible_system_image_gen` 只读 catalog item（`gateway_model_to_selector_item` 不含 api_key）→ **api_key=None, api_base=None** → ImageGenerator 回退环境变量
- catalog item 设计上不暴露凭据（安全），所以系统/团队模型无法走直连带凭据。这是架构断层。

### 2. 其它 provider / openai 兼容
- `ImageGenerator.SUPPORTED_PROVIDERS = ("volcengine","openai")`，dashscope/azure/gemini 等会被 `chat_image_gen` 直接拒（"对话生图暂不支持提供商"）
- openai 兼容路径 `_generate_openai` 同样靠环境变量回退，DB 凭据断层问题一样
- Gateway 代理路径 `proxy_non_chat_pipeline.image_generation` 正确：火山走 `volcengine_direct_image_generation`（从 DB 解析凭据+endpoint_id），其它走 LiteLLM `aimage_generation` router/direct fallback，支持所有 provider

### 3. chat 大模型路径（正常）
- `AgentLlmFacade.chat` → `_invoke_gateway_chat` → `gateway_proxy.chat_completion`，只传虚拟模型名
- 凭据/计费/日志全由 Gateway 统一处理，个人/团队/系统模型统一路径 ✅
- 这正是 image_gen 应有的架构（RFC 选项 B）

### 4. 长期方案（未执行，需单独排期）
RFC 选项 B：让 image_gen 也走 Gateway 代理
1. `GatewayProxyProtocol` 加 `image_generation` 方法
2. `GatewayBridge` 实现转发到 `ProxyUseCase.image_generation`
3. `AgentLlmFacade` 加 `image_generation` 方法
4. `chat_image_gen` / `product_image_gen_task_use_case` 改用 `AgentLlmFacade.image_generation`，删除 ImageGenerator 直连
5. 收益：所有 provider 自动支持、凭据统一、计费/日志统一、个人/团队路径统一

## 火山生图 endpoint_id 全链路传递修复（第二轮）

用户反馈凭据和模型已在远程服务器（https://gateway.giimallai.com/ai-agent/）添加，但对话生图和 Listing Studio 8 图任务仍调不通。根因是 **endpoint_id 从凭据 extra 到 ImageGenerator 的传递链路断裂**（3 层断层）。

### 断裂点
1. `RegisteredModelResolution` 缺 `extra` 字段 → `_resolution_from_row` 读了 `cred.extra` 但没传
2. `ResolvedImageGenModel` 缺 `endpoint_id` 字段 → `_resolve_personal_image_gen` 无法携带
3. `ImageGenerator.generate()` 缺 `endpoint_id_override` 参数 → 无法接收外部传入

### 修复文件（7 个）
| 文件 | 修改 |
|------|------|
| `model_catalog_port.py` | `RegisteredModelResolution` 加 `extra` 字段 |
| `sql_model_catalog.py` | `_resolution_from_row` 传 `cred.extra` |
| `chat_model_resolution_use_case.py` | `ResolvedImageGenModel` 加 `endpoint_id`；`_resolve_personal_image_gen` 用 `parse_volcengine_image_endpoint_id` 提取 |
| `image_generator.py` | `generate()` / `_generate_volcengine()` / `_resolve_volcengine_credentials()` 加 `endpoint_id_override`，优先 override 回退 config |
| `chat_image_gen.py` | 传 `endpoint_id_override=resolved.endpoint_id` |
| `listing_studio_router.py` | 传 `endpoint_id_override=resolved.endpoint_id` |
