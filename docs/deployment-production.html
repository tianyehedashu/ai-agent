AI Agent 生产环境部署指南

生产 · 阿里云 ACK 前后端分离 Nginx 反代 当前生产：`wuhan-ali` / namespace test / 公网 121.40.54.65。下文亦含裸机与 Docker Compose 参考。

概述

生产环境运行在阿里云 ACK：frontend Deployment（LoadBalancer + 内置 Nginx）反代 /ai-agent/api/ 至 backend ClusterIP；PostgreSQL / Redis 使用阿里云托管，Qdrant 在集群内。 默认路径前缀 /ai-agent（ROOT_PATH、VITE_APP_ROOT、Secret 须一致且无尾随空格）。

K8s 详细步骤：deploy/k8s/README.md · 故障排查 Skill：.agents/skills/k8s-production-debug/SKILL.md
同域反代：前端与 API 共用 LoadBalancer 入口，浏览器走相对路径，无需 CORS。

ROOT_PATH Secret：必须为 /ai-agent（无尾随空格）。用 echo -n '/ai-agent' | base64 编码；错误值会导致 OpenAPI 路径含空格、全 API 404。
勿用 /ai-agent/health 测后端——Nginx 会返回 SPA 的 index.html。

环境要求

前端服务器

  • Ubuntu 22.04+ / Debian 12+ / 任意 Linux
  • Nginx 1.20+
  • Node.js 20+（仅构建阶段，生产机可不安装）
  • 磁盘 ≥ 2 GB（静态资源 + 日志）

后端服务器

  • Ubuntu 22.04+，Docker Engine 24+（含 Compose V2）
  • 内存 ≥ 4 GB，磁盘 ≥ 20 GB
  • 开放内网端口：8000（API，不对公网暴露时可仅监听内网）

构建机（可选，与 CI 相同）

  • Git、Node.js 20+、npm
  • 用于 npm run build，产物上传至前端服务器

基础设施部署

可在后端服务器上用 Docker Compose 仅启动基础设施，或对接已有托管数据库。

方式 A：Compose 启动 DB / Redis / Qdrant

在项目根目录创建 .env.production（参考 .env.production.example），然后：

# 仅启动基础设施（不启动前后端应用容器）
docker compose -f docker-compose.prod.yml --env-file .env.production up -d db redis qdrant

方式 B：托管服务

将 backend/.env 中的连接串改为实际地址，例如：

DATABASE_URL=postgresql+asyncpg://user:pass@db-host:5432/ai_agent
REDIS_URL=redis://:password@redis-host:6379/0
QDRANT_URL=http://qdrant-host:6333

后端部署

  1. 准备环境变量

    复制模板并修改敏感项：

    cp deploy/backend.env.production backend/.env
# 必改：JWT_SECRET、POSTGRES_PASSWORD、REDIS_PASSWORD、各 LLM API Key

    前后端分离且数据库不在 Docker 网络内时，将 db、redis、qdrant 主机名改为实际 IP/域名。

  2. 构建并启动后端容器 
    # 首次：构建依赖基础镜像（若使用 Dockerfile.base 流程）
docker build -f backend/Dockerfile.base -t ai-agent-backend-base:latest backend/

# 构建并启动后端（含迁移见下文）
docker compose -f docker-compose.prod.yml --env-file .env.production up -d db redis qdrant
docker compose -f docker-compose.prod.yml --env-file .env.production build backend
docker compose -f docker-compose.prod.yml --env-file .env.production up -d backend

    若仅部署后端、不使用 Compose 中的 frontend 服务，可只 up backend 及其 depends_on 服务。

  3. 健康检查 
    curl -sf http://127.0.0.1:8000/health
curl -sf http://127.0.0.1:8000/ai-agent/health
  4. 防火墙

    后端 8000 端口仅对前端 Nginx 服务器内网 IP 开放，勿对公网全网开放（API 经 Nginx 反代对外）。

docker compose -f docker-compose.prod.yml --env-file .env.production run --rm backend alembic upgrade head

前端构建与发布

  1. 配置构建变量

    在 frontend/.env.production（或构建时 export）设置：

    # 与后端 ROOT_PATH 一致；根路径部署则设为空
VITE_APP_ROOT=/ai-agent

# 同域 Nginx 反代时留空（默认）；跨域时才设置完整 API 根
# VITE_API_URL=https://api.example.com
  2. 本地/CI 构建 
    cd frontend
npm ci
npm run build
# 产物目录：frontend/dist/
  3. 上传到前端服务器 
    # 示例：同步到 Nginx 站点目录
rsync -avz --delete frontend/dist/ deploy@frontend-host:/var/www/ai-agent/

    确保 dist/index.html 对应 URL 为 /ai-agent/（由 VITE_APP_ROOT 与 Vite base 决定）。

Nginx 配置

以下为前后端分离 + 同域反代的推荐配置。后端地址替换为实际内网 IP 或域名。

完整站点配置示例

保存为 /etc/nginx/sites-available/ai-agent.conf，并软链至 sites-enabled。

完整示例亦见 deploy/nginx/ai-agent.bare-metal.conf.example。

upstream ai_agent_backend {
    server 10.0.0.20:8000;
    keepalive 32;
}

server {
    listen 80;
    server_name app.example.com;

    root /var/www/ai-agent;
    index index.html;

    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml;

    location /ai-agent/api/ {
        proxy_pass http://ai_agent_backend/ai-agent/api/;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_read_timeout 300s;
        proxy_buffering off;
    }

    # Vite base=/ai-agent/：磁盘上 index.html 在 root 根目录，URL 带 /ai-agent 前缀
    location ^~ /ai-agent/assets/ {
        rewrite ^/ai-agent/(.*)$ /$1 break;
        expires 1y;
        add_header Cache-Control "public, immutable";
        try_files $uri =404;
    }

    location ^~ /ai-agent/ {
        rewrite ^/ai-agent/(.*)$ /$1 break;
        try_files $uri $uri/ /index.html;
    }

    location = / {
        return 302 /ai-agent/;
    }
}

根路径部署（无前缀）

若 ROOT_PATH= 且 VITE_APP_ROOT= 为空，改用：

location /api/ {
    proxy_pass http://ai_agent_backend/api/;
    # ... 同上 proxy_set_header
}
location / {
    try_files $uri $uri/ /index.html;
}

启用配置

sudo nginx -t
sudo systemctl reload nginx
常见错误：勿使用 try_files ... /ai-agent/index.html 配合 root——Vite 产物在 dist/index.html，URL 才是 /ai-agent/。须 rewrite 去前缀或改用 alias，见 deploy/nginx/README.md。
Docker 场景用 frontend/nginx.conf（上游 backend:8000）；裸机分离用 deploy/nginx/ai-agent.bare-metal.conf.example。

HTTPS

# Ubuntu / Debian + Certbot
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d app.example.com

# 生产环境建议同步修改 backend/.env
COOKIE_SECURE=true

证书自动续期由 certbot 定时任务处理。反代场景下务必设置 X-Forwarded-Proto，以便后端识别 HTTPS。

故障排查

502 Bad Gateway（Nginx）

  • 确认后端 8000 可达：curl http://<backend-ip>:8000/health
  • 检查 proxy_pass 是否包含正确的 /ai-agent/api/ 前缀
  • 查看 error.log 中 upstream 连接失败原因

404 on SPA 子路由

  • 确认 try_files 回退到 /ai-agent/index.html
  • 确认 root 目录结构与 VITE_APP_ROOT 一致

API 路径 404

  • 前后端 ROOT_PATH / VITE_APP_ROOT 必须一致
  • K8s Secret 中 ROOT_PATH 无尾随空格（见 .agents/skills/k8s-production-debug）
  • 前端请求路径为 /ai-agent/api/v1/...，不是裸 /api/v1

CORS 错误（跨域部署）

  • 在 backend/.env 设置 CORS_ORIGINS=["https://你的前端域名"]
  • 前端构建设置 VITE_API_URL=https://后端域名

迁移失败

docker compose -f docker-compose.prod.yml --env-file .env.production \
  run --rm backend alembic history
docker compose -f docker-compose.prod.yml --env-file .env.production \
  run --rm backend alembic upgrade head

国内构建慢

在 .env.production 配置 REGISTRY_PREFIX、UV_INDEX_URL、NPM_REGISTRY。
