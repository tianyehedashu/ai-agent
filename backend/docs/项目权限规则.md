# 项目权限规则



> 本文说明**谁能看到哪些页面、能使用哪些功能**，以及**不同人看到的数据是否相同**（产品 / 运营 / 测试可读）。  

> 技术实现见 [PERMISSION_SYSTEM_ARCHITECTURE.md](PERMISSION_SYSTEM_ARCHITECTURE.md) 与 [CODE_STANDARDS.md](CODE_STANDARDS.md)。  

> 自动化测试矩阵见 [tests/PERMISSION_SYSTEM_TESTS.md](../tests/PERMISSION_SYSTEM_TESTS.md)。

## 1. 角色说明



系统用三层身份组合判断权限，不要只记「是否登录」。



| 维度 | 说明 | 典型取值 |

|------|------|----------|

| **登录态** | 是否完成注册登录 | 匿名访客 / 已登录用户 |

| **平台角色** | 账号在整站的身份 | 普通用户、只读用户（预留）、**平台管理员** |

| **团队角色** | 在**当前选中的团队**里的身份 | 所有者、管理员、成员；个人空间一般为「所有者」 |



**团队内权限（AI 网关控制台）**



- **所有者 / 团队管理员**：可改团队凭据、模型、路由、预算、告警、成员等（「可管理」）。

- **成员**：可查看团队配置与日志，**不能**改团队级配置；可**创建并仅管理自己的**虚拟 Key（见 §2.2）。

- **平台管理员**：在整站额外拥有运维类菜单与跨团队统计；在某一共享团队里是否可改配置，仍看该团队内的角色（平台身份不自动等于团队所有者）。

## 2. 数据可见性（不同的人看到的数据是否一样）



**结论：不一样。** 系统按「你是谁、属于哪些团队、当前选中的是哪个团队」过滤列表与详情；没有「全站所有人看到同一份业务数据」的设计（平台管理员在部分运维场景除外，见 §7、§9）。



### 2.1 三条原则



1. **工作区（团队 / `tenant_id`）是多数数据的边界**  

   Agent、记忆、Listing/视频任务、网关团队配置、调用日志等，行级过滤依赖 `PermissionContext.team_ids`（用户加入的所有团队，含个人团队）。未加入的工作区：列表为空或访问被拒绝。



2. **个人空间也是独立工作区**  

   注册后自动有一个「个人团队」（`Team.kind=personal`）。个人凭据、个人模型挂在个人团队下，**其他用户看不到**；与后来加入的「共享团队」互不混用，除非在界面里切换到那个团队。



3. **当前选中的团队决定网关里「团队/共享」看到什么**  

   在 AI 网关顶部切换团队后，凭据、模型、路由、日志、预算等**团队维度**列表会随之切换；选团队 A 时看不到团队 B 的配置。



### 2.2 例外：会话不按「团队成员」共享



| 规则 | 说明 |

|------|------|

| **会话仅 personal tenant** | 会话行的 `tenant_id` 只对应创建者的个人工作区（含匿名 shadow user 的个人团队），**不会**因两人同在共享团队 T 而互相看到对方会话。 |

| **列表过滤** | 会话列表按 personal `tenant_id` 查询，**不用** `team_ids` 做宽过滤（避免误把共享团队会话混在一起）。 |

| **详情鉴权** | REST `/sessions/{id}`、Chat、**检查点**（`/chat/checkpoints/*`）统一走 `SessionUseCase.assert_session_accessible`（personal tenant 等值）。 |

| **平台管理员** | `role=admin` 可按会话 ID **读取任意会话详情**（运维排障）；**列表仍仅本人 personal 会话**，不能靠列表枚举全站会话。 |



### 2.3 按数据类型：谁能看到什么



| 数据类型 | 可见范围 | 补充说明 |

|----------|----------|----------|

| **聊天会话与消息** | **仅本人**（personal `tenant_id`） | 见 §2.2；匿名访客仅 Cookie 绑定的 shadow 用户会话 |

| **Chat 检查点** | 与同会话一致 | 须先通过会话鉴权再读 Redis 中的检查点状态；他人仅有 `checkpoint_id` 不能绕过会话权限 |

| **Agent** | 本工作区成员（`tenant_id ∈ team_ids`） | 若 **公开**，同工作区内其他成员可**查看**，不可代为修改 |

| **记忆（Memory）** | 本工作区内 | 按 `tenant_id` + `team_ids` 过滤，与会话「仅本人」规则不同 |

| **网关 · 个人凭据 / 个人模型** | 仅本人（个人工作区） | 与「团队/共享」页签无关 |

| **网关 · 团队凭据 / 模型 / 路由 / 预算 / 告警** | 当前选中团队 + 成员身份 | 须为团队成员；列表合并 **系统级** 全局模型/路由 |

| **网关 · 虚拟 Key** | **仅创建者本人** | 团队 owner/admin **不能**查看或代用成员的 Key |

| **网关 · 调用日志（workspace 聚合）** | 当前团队 | **成员**在未指定 vkey 时：仅看自己创建的非系统 vkey 相关行 + **本人**平台入站（`user_id` 匹配）行；owner/admin 看全团队 |

| **网关 · 调用日志（单条详情）** | 同上 + 成员二次过滤 | 成员打开他人 vkey 或他人入站行 → 拒绝（403 语义） |

| **设置 · 平台 API 密钥** | 仅本人 | 与网关虚拟 Key 不同类 |

| **MCP 服务器（用户自建）** | 仅本人工作区 | 每人维护自己的连接配置 |

| **MCP HTTP 调用** | API Key + scope | 须带 `mcp_*` 等作用域；通过后安装与登录一致的 `PermissionContext` |

| **视频 / Listing 任务** | 本工作区内 | `tenant_id` 归属，非「全站 user_id 列」 |

| **网关 · 平台统计** | 仅平台管理员 | 跨工作区运维统计 |

| **对象存储配置** | 全站一份 | 仅平台管理员可改 |



### 2.4 常见对比（便于测试与客服）



| 场景 | 用户 A 能看到 | 用户 B 能看到 | 是否相同 |

|------|----------------|----------------|:--------:|

| A、B 无共同工作区 | 各自个人空间的数据 | 各自个人空间的数据 | 否 |

| A、B 同在共享团队 T | 个人空间 + 团队 T 的网关团队数据 | 同上 | 部分相同（**不含**对方会话/Agent 私有行，除非 Agent 公开且同 tenant） |

| B 在 T1、T2，网关切到 T1 | — | T1 的团队凭据/日志等 | 切换团队后列表**会变** |

| B 为团队 T 的**成员** | — | T 配置只读；日志子集；vkey 仅自己的 | A（owner）**看不到** B 的 vkey |

| 平台管理员 vs 普通用户 | 个人 + 已加入团队 + 运维菜单；**可按 ID 读任意会话** | 个人 + 已加入团队 | 管理员**无**全站会话列表 |



### 2.5 与「菜单权限」的关系



- **菜单/按钮**：决定你能不能进入某功能、能不能点「保存」「删除」。  

- **数据范围**：决定进入后列表里**有哪些行**、点开详情是否报错。  



例如：两人都能看到「AI 网关 → 调用日志」，但若不在同一团队或选中团队不同，**日志内容不同**；虚拟 Key 列表**按创建者隔离**。



前端 Gateway 写按钮与 `canWrite`（团队 owner/admin 或平台 admin）对齐，见 `frontend/src/hooks/use-gateway-permission.ts`。**个人凭据**（`/gateway/credentials` 个人 Tab）只要求登录，**不能**用团队 `canWrite` 判断。

## 4. 匿名访客



**可以使用**



- 聊天、会话列表与历史（personal shadow 工作区）

- Agents 列表（受 `team_ids` 数据范围限制）

- 用户 MCP 配置、系统 MCP（查看与复制接入说明）

- 视频任务、Listing 创作

- 设置中的通用项（如主题）



**不能使用或受限**



- 侧栏不出现 **AI 网关**（须登录）

- 设置中 **API 密钥、账户信息** 的保存与删除账户

- 不能访问其他注册用户或团队的数据

## 5. 已登录普通用户



注册登录后默认拥有 **个人团队（工作区）**，在个人团队内一般为 **所有者**。



### 5.1 设置



| 页签 | 功能 |

|------|------|

| 通用 | 主题、通知等 |

| API 密钥 | 创建、查看、吊销**平台 API 密钥**（产品 API；≠ 网关虚拟 Key） |

| MCP 服务器 | 管理自己的 MCP 连接配置 |

| 账户 | 修改昵称、厂商用户 ID 等 |



说明：凭据/个人模型在 **AI 网关 → 凭据 / 模型** 的「个人」页签；旧设置页链接会重定向到网关。



### 5.2 AI 网关（须登录）



进入 **AI 网关** 后，左侧为二级菜单。「可管理」= 当前团队 owner/admin，或平台 admin。



| 子菜单 | 所有人 | 可管理 | 仅平台管理员 |

|--------|:------:|:------:|:------------:|

| 概览 | 查看仪表盘（请求/Token/成功率/延迟）；**累计成本**仅 admin+；**套餐毛利**仅共享团队 admin+ / 平台 admin | — | — |

| 调用指南 | 阅读文档 | — | — |

| 虚拟 Key | 仅**自己创建**的 Key | 团队成员可新建；仅创建者可吊销自己的 | — |

| 凭据 | 查看个人 + 当前团队 | 个人与团队增删改、探测、导入 | **系统级**凭据 |

| 模型 | 查看个人 + 团队模型 | 注册、编辑、删除、连通性测试 | — |

| 定价目录 | 「我的价格」 | 「下游售价」（团队 admin+） | 「上游成本」 |

| 虚拟路由 | 查看 | 增删改 | — |

| 预算配额 | 查看 | 创建、调整 | — |

| 调用日志 | 查看（成员见 §2.3 子集） | — | — |

| 告警规则 | 查看 | 增删改 | — |

| 团队成员 | 查看 | 新建团队、增删成员；所有者可删非个人团队 | 亦可删团队 |

| 平台统计 | — | — | 查看 |



**聊天里选模型**：`GET /gateway/models/available` = 系统可见目录 + **个人**已注册模型（不随团队 Tab 切换团队模型列表）。



### 5.3 其他功能模块



| 模块 | 已登录普通用户 |

|------|----------------|

| **Agents** | 在本工作区内创建、编辑、删除；公开 Agent 可被同 tenant 成员查看 |

| **聊天** | 建会话、对话；检查点与时间旅行调试受会话权限约束 |

| **MCP 服务器** | 管理自己的 MCP 配置 |

## 6. 共享团队中的「成员」



用户被邀请进**共享团队**且角色为 **成员** 时，在网关里切换到该团队后：



| 能力 | 成员 | 团队管理员 / 所有者 |

|------|:----:|:------------------:|

| 查看概览、指南、日志（子集）、模型、路由、凭据、预算、告警 | 是 | 是 |

| 修改团队凭据、团队模型、路由、预算、告警 | 否 | 是 |

| 虚拟 Key | 仅自己创建、自己使用 | 仅自己创建、自己使用 |

| 定价「下游售价」 | 不可见 | 可见可改 |

| 添加/移除团队成员、新建共享团队 | 否 | 是 |

## 7. 平台管理员



在**已登录普通用户的全部能力**之外，额外包括：



| 类别 | 功能 |

|------|------|

| **平台运维** | **对象存储**配置 |

| **网关跨团队** | **平台统计** |

| **网关配置** | 编辑 **系统级** 凭据；`catalog/reload-from-config` |

| **定价** | **上游成本**、全局定价 |

| **系统 MCP** | 动态工具与提示词 |

| **数据层（须知晓）** | `PermissionContext.is_admin` 时仓储 **不过滤** tenant（可跨 tenant 读 Agent 等）；**会话列表仍 personal**；**可按 ID 读任意会话** |



平台管理员**不会自动**成为每个共享团队的所有者；在某团队内能否改配置，仍取决于该团队内角色。

## 8. 只读平台用户（viewer）



平台 **viewer** 账号在 **AI Gateway 管理面**为只读：后端 `resolve_current_team` 拒绝非 GET/HEAD/OPTIONS；前端 `useGatewayPermission` 的 `canWrite` 恒为 `false`（虚拟 Key 等仍用 `canManageKeys = isMember && !isPlatformViewer`）。

## 9. 安全边界、测试要点与常见漏洞



本节汇总**文档易漏**的实现约束与审计时应测的 IDOR 面。



### 9.1 必须遵守的实现约束（隐藏规则）



| 约束 | 原因 |

|------|------|

| **`PermissionContext` 单一装配** | 仅 `PermissionContextComposer` + 认证依赖 + 权限中间件可 `set_permission_context`（架构测试白名单） |

| **SSE Chat/Resume** | 流式生成器内用 `PermissionContextComposer.scoped` 恢复上下文，禁止裸改 ContextVar |

| **会话鉴权统一入口** | 禁止再使用已删除的 `check_session_ownership`；Chat 检查点与 Session REST 同策略 |

| **检查点先鉴权后加载** | `resolve_session_id` → `assert_session_accessible` → 再 `get_or_raise` |

| **租户资源显式检查** | Agent/Memory 等用 `check_tenant_access` / `check_tenant_access_or_public` |

| **Gateway 匿名 401** | `/api/v1/gateway/*` 与 `/v1/*` 代理不接受匿名 Cookie 会话 |

| **MCP 路径** | API Key 校验 scope 后必须 `composer.install`，否则仓储 `tenant_id` 无法解析 |



### 9.2 常见漏洞模式与防护



| 风险 | 错误做法 | 正确做法 / 现状 |

|------|----------|-----------------|

| **IDOR（会话）** | 仅校验 `session_id` 存在 | `assert_session_accessible`（personal tenant） |

| **IDOR（检查点）** | 仅凭 `checkpoint_id` 读 Redis | 先解析 `session_id` 并做会话鉴权 |

| **IDOR（Agent）** | `get_by_id` 后不校验 | `check_tenant_access` 或 public 策略 |

| **团队上下文混淆** | 用 `X-Team-Id` 访问他人工会话 | 会话与团队选择**解耦** |

| **成员日志越权** | workspace 列表不过滤 member | `usage_log_visibility` 策略 + axis `member_user_id` |

| **vkey 代管** | owner 查看成员 sk-gw | `virtual_key_access` 仅创建者 |

| **测试缺 team_ids** | 单测 `PermissionContext` 无 `team_ids` | 使用 `tests/helpers/permission_context.py` |



### 9.3 已知产品/技术缺口（非漏洞但需知晓）



| 项 | 状态 |

|----|------|

| **viewer 角色** | Gateway 管理面已只读（API + `canWrite`）；其它域若需只读需单独加规则 |

| **平台 admin 仓储宽读** | `is_admin` 绕过 tenant 过滤；依赖人工流程，非全站会话列表 |

| **Legacy `Owned*` / `check_ownership`** | 已从代码库移除（2026-05） |

| **HTTP 字段 `team_id`** | 与 `tenant_id` 并存，用于展示/计费，**不是**第二套归属列 |

| **E2E 依赖真实 Gateway** | 无 Key 时部分用例 skip |
