Gateway 费用计算与 LiteLLM 集成

说明 LiteLLM 如何计算单次调用的 USD 成本，以及本仓库 AI Gateway 如何在 LiteLLM 之上实现「上游成本 / 下游收入」分离、持久化与展示。

上游 cost_usd 下游 revenue_usd 库内 USD · 展示 CNY/USD

域边界详见 AI_GATEWAY_DOMAIN_ARCHITECTURE.md 第 4.8 节 · 源文档 GATEWAY_PRICING_AND_LITELLM_COST.md

1. 概念对照

概念LiteLLM本项目
单价注册表 litellm.model_cost / register_model() upstream_model_pricing + sync_to_litellm_registry()
单次调用成本 response_cost（_hidden_params） 请求日志 cost_usd（上游成本）
对客户计费 margin 叠在一条 cost 上（见下文） downstream_model_pricing → revenue_usd
用量字段 usage.prompt_tokens / completion_tokens 同左 + gateway_request_logs
估算 completion_cost() PricingService.calculate()；前端 estimateUsageCostDisplay()

币种约定：数据库与结算逻辑一律 USD；API / 前端经 MoneyProjector + FxRatePort 投影为 CNY 或 USD 展示。

LiteLLM 是否「天然」上下游分开计费？

结论：不完全是
LiteLLM 强在 一条成本线（response_cost）+ 可选 Provider Margin （百分比或固定费叠在 base cost 上）。本项目则是 两张价目表，日志中同时存 cost_usd 与 revenue_usd，支持 team / entitlement / 按模型定价。

LiteLLM

  • base cost → discount → margin → 一个最终 cost
  • 粒度多为 provider / global
  • Lago 等外接账单用 response_cost

本项目 Gateway

  • cost_usd ≈ 厂商成本（LiteLLM）
  • revenue_usd = 下游价 × token
  • 毛利 = revenue − cost；管理员可见上游

2. LiteLLM 费用计算机制

2.1 内置价目表 model_cost

全局字典，键为模型 ID，值为单价字段（与 OpenAI 风格对齐）：

字段含义
input_cost_per_token输入 token 单价（USD）
output_cost_per_token输出 token 单价（USD）
cache_creation_input_token_costPrompt 缓存写入（可选）
cache_read_input_token_costPrompt 缓存读取（可选）
input_cost_per_image 等图像 / 音频等非 token 计费

虚拟模型名（如 team/foo）常不在内置表中，需 register_model()。

2.2 注册与覆盖

import litellm

litellm.register_model({
    "my-virtual-model": {
        "input_cost_per_token": 0.00000015,
        "output_cost_per_token": 0.0000006,
    }
})

本项目在 PricingService.sync_to_litellm_registry() 统一注册；Router deployment 不必再带单价。

2.3 响应阶段 response_cost

response._hidden_params["response_cost"]  # float，USD

同时进入 standard_logging_object["response_cost"]。等价公式：

cost = Σ (token_count × 单价) + 按请求固定费（若配置）

2.4 事后计算 completion_cost()

from litellm import completion_cost
usd = completion_cost(completion_response=response, model="gpt-4o")
调用点策略
custom_logger._calc_cost _hidden_params → standard_logging_object → completion_cost()
proxy_use_case._calc_cost 直接 completion_cost()（预算 Redis）

2.5 标准日志与排障

StandardLoggingPayload 含 response_cost、model_map_information、custom_pricing。 与厂商账单不一致时： 先核对 token，再核对单价映射。

4.4 域内计价公式

total = input_cost_per_token * input_tokens
      + output_cost_per_token * output_tokens
      + cache_creation_input_token_cost * cache_creation_tokens
      + cache_read_input_token_cost * cache_read_tokens
      + per_request_usd * requests

7. 运维与排障

7.1 成本为 0 或偏差大

  • 检查 upstream_model 是否已 register_model
  • 对比 pricing_snapshot.response_cost 与 cost_usd
  • 补全上游价后重启 sync
  • 厂商账单：先 token，后单价

7.2 收入 = 成本

未配下游价，或 mirror / upstream_passthrough；查 hit_chain 与生效期。

7.3 套餐内调用

gateway_billing_package=entitlement：日志双零；报表需区分日志费用与套餐额度。

8. 计费这块，还有哪些值得优化？

下面按「先修会算错钱的 → 再省运维力气 → 最后做体验」排列。 说的是现状里比较实在的改进点，不是画大饼。

  • 优先 预算扣款和日志金额，最好同源

    现在：预算走 proxy 里的 completion_cost()； 日志里的 cost_usd 优先用 LiteLLM 写在响应里的 response_cost。 多数情况一样，但边缘场景（虚拟模型没注册、流式没带好 usage）可能预算扣了 A、日志记了 B。

    建议：抽一个「算上游成本」的小函数，预算和 custom_logger 共用；算不出来时统一打标 pricing_snapshot.cost_estimated=true，方便对账。

  • 优先 上游单价和 LiteLLM 注册键要对齐

    LiteLLM 按 upstream_model 查价。键对不上时，response_cost 经常是 0， 看起来像「这次调用免费」，其实是没价目。

    建议：启动/改价后做一次自检（哪些模型有 deployment 但没有 upstream 行）；管理面保存上游价时提示「将注册为 xxx 键」。

  • 优先 流式请求的成本别含糊

    非流式响应能拿到完整 response_cost；流式主要靠末帧 usage 再调 completion_cost， 和回调里那条链路不完全一样，长连接、中断、只回了半截 usage 时更容易偏。

    建议：以 callback 落库为准做「最终账单」；Proxy 侧预算可先按 token 粗估，回调后再修正（或流式也尽量透传 LiteLLM 的 hidden cost）。

  • 中等 上游价目别全靠手填

    现在主要靠 app.toml 同步 + 管理员维护。LiteLLM 官方价表会更新，新模型也层出不穷。

    建议：定期从 LiteLLM 内置 model_cost 拉增量（类似官方 POST /reload/model_cost_map 思路），人工只改「和厂商谈好的特价」； 变更留 effective_from，避免历史日志对不上。

  • 中等 每次请求都查下游价，可以缓存

    attach_downstream_pricing_metadata 每次调用会解析 team / 套餐 / 模型价，要打数据库。 价目表变更不频繁，高 QPS 时这是纯开销。

    建议：按 (team_id, gateway_model_id, entitlement_plan_id) 短 TTL 缓存解析结果；改价时发事件清缓存。

  • 中等 套餐内「日志为 0」要让人看得懂

    命中 entitlement / provider 套餐时，cost_usd 和 revenue_usd 都记 0， 但预算、套餐额度可能还在扣。运营看报表容易问：「为什么用量有、钱没有？」

    建议：日志里固定带 billing_package +「影子金额」（仅管理员可见的上游成本），或单独「套餐消耗」报表，和「按量计费」分开。

  • 中等 图 / 视频 / embedding 等非 chat 能力

    域内公式以 token × 单价 为主；复杂计费（按张图、按秒、按次）在 extra 里， 下游 revenue_usd 是否每次都按同一套规则算齐，需要逐能力验收。

    建议：按 capability 列一张「是否已对齐 LiteLLM 计价」清单；缺的支持 per_request_usd 或单独 settlement 分支。

  • 体验 展示汇率别永远写死

    现在用静态 gateway_fx_usd_cny，前端 CNY 展示好看，但和财务月结汇率可能对不上。

    建议：配置日汇率或「仅展示用 / 结算用」分离；日志 pricing_snapshot.fx_rate_used 已有，可再标明来源与日期。

  • 体验 下游价配置与排障

    收入等于成本，多半是 mirror / 没配下游价；要靠 hit_chain 和 DBA 查表，对非开发不友好。

    建议：管理面「这条请求为什么是这个价」可视化；bulk mirror 后预览毛利；生效期临近发提醒。

  • 体验 前端「预估费用」和账单对齐

    Playground / 我的价格页用「每百万 token」估算；真实账单走 callback 结算，含 cache、套餐、按次费等。

    建议：估算接口直接复用 PricingService.calculate 或返回「估算 / 实际可能偏差」说明，减少用户信任落差。

  • 体验 对账与导出

    排障步骤已写在 §7，但缺「一键导出团队某月 cost / revenue / 毛利」和与厂商账单 diff 的小工具。

    建议：按 team + 月聚合（rollup 已有基础的话接上）；对标 LiteLLM 排障：先 token、后单价，自动生成差异 Top 模型。

和 LiteLLM 自带 margin 的关系
我们刻意用两张价目表做上下游，而不用 LiteLLM 的「在 cost 上加 margin」。 运维上要避免：又在 LiteLLM 配 margin、又在下游表加价，重复计费。 文档和配置检查里写死一句即可。
